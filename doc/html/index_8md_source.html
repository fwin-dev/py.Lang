4 # Summary of functionality
5 
6 This package provides a lot of miscellaneous things that probably should have been included in python's built-in
7 library but weren't, including:
8 
9 * [FuncTools](#functools): Various function/method utilities/tools
10  * `FuncTools.Abstraction.abstractmethod`: An abstract method that can be used with `@classmethod`
11  * `FuncTools.timeIt`: Timing execution of a function with a decorator
12  * `FuncTools.getArgs`: A way to get the argument names and values of a function without using `**kwargs` or `*args`
13 * [ClassTools](#classtools): Various utilities/tools for working with classes. Also includes class patterns.
14  * `ClassTools.vars`: Getting the variables of a class instance with `__slots__`
15  * `ClassTools.Patterns`: A package with implementations of [class patterns](http://en.wikipedia.org/wiki/Software_design_pattern)
16  * `ClassTools.Patterns.RegisteredInstances`: Easily keep track of all instances of a class
17  * `ClassTools.Patterns.Singleton`: Class bases (using metaclass) that implement the singleton pattern
18  * `ClassTools.Patterns.Multiton`: Class bases (using metaclass) that implement the multiton pattern
19  * `ClassTools.Patterns.StartEndWith`: Automatic support of the `with` statement by implementing only a start and end method
20 * [PyPkgUtil](#pypkgutil): Various module and package utilities/tools (Is a module built in? In what file is it? etc.)
21 * [Struct](#struct): Implementation of various structures to hold data
22  * `Struct.LIFOstack`: LIFO/Stack
23  * `Struct.FrozenDict`: Frozen dictionary
24  * `Struct.OrderedSet`: Ordered set
25 * [Iter](#iter)
26  * `Iter.PeekableIterable`: A peekable iterator
27 * [Diff](#diff)
28  * `Diff.SequenceMatcher`: An improved differ, based on python's builtin `difflib.SequenceMatcher`
29 * [Concurrency](#concurrency): Unified API for locks, semaphores, etc., along with some useful tools/utilities
30  * `Concurrency.Multiprocessing`: For dealing with multiple python processes
31  * `Concurrency.Multiprocessing.decorators.processify`: Run a function in a separate process
32  * `Concurrency.decorators.useLock`: Surround an entire function's execution in a lock
33  * `Concurrency.Threading`: A lock and semaphore using standard python threads
34  * `Concurrency.FileSystem.FileLock_ByFCNTL`: A lock using unix FCNTL file locking
35 * [Terminal](#terminal): Utilities for improving terminal interaction with the user
36  * `Terminal.askYesNo`: A simple way of asking user to respond with yes or no
37  * `Terminal.FormattedText`: Color text in the terminal. Also can do bold, underline, etc. depending on the terminal.
38  * `Terminal.Table`: Nicely prints column+row data in the terminal.
39  * `Terminal.ArgParser`: An improved ArgParser for parsing command line arguments
40 * [Events](#events): Easy event handling with subscriptions+callbacks, including an API for event logging
41  * `Events.Proxy`: Provides a super easy API for event subscription and callbacks
42  * `Events.Logging`: Skeleton classes for logging events to different destinations
43 * [DebugTracer](#debugtracer): A poor man's debug tracer when a better tracer isn't available (for example, when running a python script over ssh without remote debugging)

77 ### Getting a function's arguments
78 
79 Python provides `**kwargs` and `*args` to get a dictionary or list of a function's arguments, but this makes it hard
80 for IDEs and documentation generators to determine all possible arguments to the function. As an alternative to
81 `**kwargs` or `*args`, you can specify all arguments explicitly and then use the `getArgs()` method:
82 
83  from Lang.FuncTools import getArgs
84  def myFunc(arg1, arg2):
85  allKwargs = getArgs()
86 
87 This method uses python's built in `inspect` module to go up the stack and inspect arguments.
88 
89 In the case above, `allKwargs` will be a dict of parameter names and associated values, similar to as if `**kwargs` was used.
90 
91 To get a list of positional values instead, similar to `*args`, use `getArgs(useKwargFormat=False)`.
92 
93 To get both kwargs and args, use:
94 
95  args, kwargs = getArgs(useKwargFormat=None)
96 
97 * Note that `cls` and `self` are automatically ignored for class methods and instance methods.
98 * Note that if there is a question of whether an argument is an arg or a kwarg, then kwarg is preferred.

111 ### Patterns
112 
113 #### Keeping track of class instances
114 
115 A common pattern in programming is the need to access all instantiations of a class for some reason. In static languages, this is
116 often done in a factory, but in python, we can use metaclasses to implement this, and it gives a nicer interface for the programmer.
117 For example:
118 
119  from Lang.ClassTools.Patterns import RegisteredInstances
120 
121  class Foo(RegisteredInstances):
122  def __init__(self, value):
123  super(Foo, self).__init__()
124  self.value = value
125 
126  a = Foo("asdf")
127  b = Foo("kjlh")
128 
129 Now we can use the following to get an OrderedSet of all instances created:
130 
131  fooInstances = Foo.getAllInstances()
132 
133 The `RegisteredInstances` superclass also aims to support subclasses of your `Foo` in a nice way. For example, if we additionally define:
134 
135  class Bar(Foo):
136  pass
137 
138 And then instantiate:
139 
140  c = Bar("iuhl")
141 
142 We'll get the following results:
143 * A call to `Foo.getAllInstances()` will return an OrderedSet of all 3 instances (2 of `Foo`, 1 of `Bar`) instantiated up to this point.
144 * A call to `Foo.getAllClasses()` will return an iterable of 2 classes, `Foo` and `Bar`.
145 * A call to `Bar.getAllInstances()` will return an OrderedSet of 1 instance of `Bar`.
146 * A call to `Bar.getAllClasses()` will return an iterable of 1 class, `Bar`.
147 
148 #### Singleton pattern
149 
150 A well known pattern is [the singleton pattern](http://en.wikipedia.org/wiki/Singleton_pattern). There are two implementations available
151 in this package, depending on wanted behavior when your class's constructor is (incorrectly) called multiple times (aka `duplicate` instances);
152 one implementation will raise an exception for the second instantiation attempt, and the other will simply ignore and discard the second
153 attempt, and give the originally instantiated object back. Let's see these in action:
154 
155  from Lang.ClassTools.Patterns import Singleton_OnDupRaiseException
156 
157  class Foo(Singleton_OnDupRaiseException):
158  def __init__(self, value):
159  super(Foo, self).__init__()
160  self.value = value
161  a = Foo(1)
162  b = Foo(2) # an exception is raised
163 
164 And the other singleton implementation:
165 
166  from Lang.ClassTools.Patterns import Singleton_OnDupReturnExisting
167 
168  class Foo(Singleton_OnDupReturnExisting):
169  def __init__(self, value):
170  super(Foo, self).__init__()
171  self.value = value
172  a = Foo(1)
173  b = Foo(2) # b actually holds the Foo(1) instance, and Foo(2) is not created
174 
175 Note: Make sure the singleton superclass is the **first** in your MRO heirarchy.
176 
177 The following methods would be available on `Foo`, similar to the "instance tracking" pattern, above. (See there for more details.)
178 Differences are noted here:
179 * `Foo.getAllClasses()`: Same behavior as above.
180 * `Foo.getAllInstances()`: Will return a dictionary with the key being the class, and the value being the singleton instance.
181 
182 #### Multiton pattern
183 
184 There are two implementations of [the multiton pattern](http://en.wikipedia.org/wiki/Multiton_pattern) aka "registry of singletons",
185 similar to the singleton pattern above. Although wikipedia says a multiton is simply a singleton that keeps track of instances via key+value
186 storage, this is very vague. In the py.Lang multiton implementation, each instance of the class must not be equal to another instance of the class
187 (hence the name `Multiton_OneEquivalentInstance`), else either an exception is raised, or the existing instance, which is equal, is returned
188 (hence the names `OnDupRaiseException`/`OnDupReturnExisting`, same as the singleton pattern). The uniqueness of the instance can be defined by
189 the `__eq__` method. However, unlike the singleton pattern, since the instance must be created in order to compare using `__eq__`, be careful
190 because the object's `__init__` method will always be called, regardless of whether the new instance is equal to an already-created instance.
191 This could introduce an unintended side effect if not known in advance. If the new instance is not in fact valid, then it will be deleted
192 (actually garbage collected because it will fall out of scope) after `__init__` and `__eq__` have been called.
193 
194 The classes which can be inherited from are `Multiton_OneEquivalentInstance_OnDupRaiseException` and `Multiton_OneEquivalentInstance_OnDupReturnExisting`.
195 Here is an example 2 instances allowed because the comparison is done by memory address, not using `__eq__`:
196 
197  class Foo(Multiton_OneEquivalentInstance_OnDupRaiseException):
198  def __init__(self, value):
199  super(Foo, self).__init__()
200  self.value = value
201  a = Foo(3)
202  b = Foo(3)
203  self.assertNotEqual(id(a), id(b))
204 
205 And here's an example using `__eq__`:
206 
207  class Foo(Multiton_OneEquivalentInstance_OnDupRaiseException):
208  def __init__(self, value):
209  super(Foo, self).__init__()
210  self.value = value
211  def __eq__(self, other):
212  return isinstance(other, self.__class__) and self.value == other.value
213  a = Foo(3)
214  b = Foo(3) # will raise a DuplicateInstanceException

216 #### StartEndWith pattern
217 
218 This pattern provides some nice functionality that just implementing `__enter__` and `__exit__` doesn't provide.
219 
220 ##### Needing to manually enter and exit
221 
222 Find yourself implementing `__enter__` and `__exit__`, but still needing to manually call the entrance method and exit method sometimes?
223 Yes, you can manually call `__enter__` or `__exit__` without using `with`, or you can define other methods like `start` and `end`
224 which call `__enter__` and `__exit__`. The StartEndWith pattern gives you those nice looking `start` and `end` methods. You can even rename the
225 methods by passing different method names into the constructor:
226 
227  class Foo(StartEndWith):
228  def __init__(self):
229  super(Foo, self).__init__(methodName_start="begin", methodName_end="end")
230  def _start(self, *args, **kwargs):
231  <your code to start something>
232  def _end(self, *args, **kwargs):
233  <your code to end something>
234 
235  a = Foo()
236  a.begin()
237  <do something>
238  a.end()
239 
240 ##### Telling whether you've began or not
241 
242 StartEndWith defines an `isActive()` method, which returns whether or not you're inside the `with` statement (or have called `start()`).
243 Continuing from the example above:
244 
245  print(a.isActive()) # False
246  a.begin()
247  <do something>
248  print(a.isActive()) # True
249  a.end()
250  print(a.isActive()) # False
251 
252 Or, using `with`:
253 
254  with Foo() as a:
255  print(a.isActive()) # True
256 
257 ##### Starting more than once without ending
258 
259 If you'd like to be able to start something more than once without ending it, use `allowStartWhileRunning=True`. (Otherwise, an exception will be raised.)
260 
261  class Foo(StartEndWith):
262  def __init__(self):
263  super(Foo, self).__init__(allowStartWhileRunning=True)
264  def _start(self, *args, **kwargs):
265  <your code to start something>
266  def _end(self, *args, **kwargs):
267  <your code to end something>
268 
269  a.start()
270  a.start()
271  # this is also valid
272  a.end() # only need to call once
273 
274 ##### Disposable resource
275 
276 If you only want to be able to call your start and end code (either via explicit calls or using `with`) once and never again, use `useOnce=True`:
277 
278  class Foo(StartEndWith):
279  def __init__(self):
280  super(Foo, self).__init__(useOnce=True)
281  def _start(self, *args, **kwargs):
282  <your code to start something>
283  def _end(self, *args, **kwargs):
284  <your code to end something>
285 
286  a = Foo()
287  a.begin()
288  <do something>
289  a.end()
290  a.begin() # exception is raised
291 
292  with a: # exception is raised
293  <do something>

398 ### Peekable iterator
399 
400 Want to use an iterator for RAM usage concerns, but need to know the next element sometimes without advancing the
401 iterator? Then this "peekable iterator" implementation is for you! Just wrap any iterable inside a `PeekableIterable`.
402 That inner iterable is most likely a pure iterator itself (such as a generator), but it could also be anything that
403 implements `__iter__`.
404 
405  from Lang.Iter import PeekableIterable
406  def asdf():
407  for i in range(0,10):
408  yield i
409  nums = PeekableIterable(asdf())
410  print(nums.next()) # advances the iterator
411  print(nums.peek()) # peeks without advancing
412  print(nums.hasNext()) # checks if there's a next element

414 ## <a name="diff"></a>Diff
415 
416 ### An improved differ
417 
418 This differ builds upon `difflib.SequenceMatcher` which can diff any python objects with `__eq__` implemented, contained
419 within a list, tuple, etc. However, the built in class only provides methods for getting matching sets of indices which
420 refer to matching elements, leaving you to infer which indices don't match. It also doens't give you direct access to
421 elements that do or don't match. This differ adds all of that functionality. It also fixes:
422 
423 * A bug: In the built in differ, subsequent calls to `get_matching_blocks()` will return results in a different format due to caching
424 * Calculating the similarity ratio: The built in differ calculates this ratio taking both diff sides into account, but what's usually wanted is how much one side is similar/different compared to the other side: `(1 - ratio) / 2 + ratio`
425 
426 Examples:
427 
428  from Lang.Diff import SequenceMatcher
429  diff = SequenceMatcher(tuple("aebcdef"), tuple("abbcdgef"))
430  print(list(diff.get_matching_blocks()))
431  print(list(diff.get_mismatching_blocks()))
432  print(list(diff.get_matching_elems()))
433  print(list(diff.get_mismatching_elems()))
434 
435  Prints the following:
436 
437  [BlockMatch(a={index=0,size=1}, b={index=0,size=1}), BlockMatch(a={index=2,size=3}, b={index=2,size=3}), BlockMatch(a={index=5,size=2}, b={index=6,size=2})]
438  [BlockMismatch(a={index=1,size=1}, b={index=1,size=1}), BlockMismatch(a=None, b={index=5,size=1})]
439  [ElemMatch(a=('a',), b=('a',)), ElemMatch(a=('b', 'c', 'd'), b=('b', 'c', 'd')), ElemMatch(a=('e', 'f'), b=('e', 'f'))]
440  [ElemMismatch(a=('e',), b=('b',)), ElemMismatch(a=None, b=('g',))]
441 
442 More functions are available, including:
443 
444 * `getmatching()` and `getmismatching()`
445  * These return structures with `.block` and `.elems` attributes containing both block indices and the elems which the block refers to
446 * `get_matching_elems_useOnce()` and `get_mismatching_elems_useOnce()`
447  * These are the same as `get_matching_elems()` and `get_mismatching_elems()` except that they are generators instead of functions returning a list

449 ## <a name="concurrency"></a>Concurrency
450 
451 The `Concurrency` package provides a unified API for locks and semaphores, in addition to some useful utilities.
452 
453 ### @processify
454 
455 Using this function decorator will automatically cause the function to run inside a new python process:
456 
457  from Concurrency.Multiprocessing.decorators import processify
458 
459  @processify
460  def foo():
461  <do something>
462 
463 * Note that the code is not run in parallel to the current process, so this is not for gaining any speed.
464 * Note that every argument and the return value must be picklable.
465 
466 ### The unified API
467 
468 Different threading APIs and similar will generally have `lock` and `release` methods, along with possibly some other methods and functionality.
469 A unified API was made in order to smooth over these differences and fill in functionality that was missing. This includes:
470 
471 * Standardized method names
472 * Standardized parameters for `lock` and `release` methods
473 * Standardized parameters for lock/semaphore constructor
474 
475 #### @useLock
476 
477 Using this function decorator will automatically cause a lock to be acquired before executing the function, and released after the function exits:
478 
479  from Concurrency.decorators import useLock
480 
481  lockInstance =
482  @useLock(lockInstance)
483  def foo():
484  <do something>

486 ## <a name="events"></a>Events
487 
488 Easy event handling with subscriptions+callbacks, including an API for event logging.
489 
490 ### Events.Proxy
491 
492 For general event handling involving event subscription and listeners, there is a proxy API. The proxy receives an event
493 (via a method call) and then calls any subscribers (aka receivers) to the event. In this implementation, a subscriber
494 subscribes to all events, but only chooses to implement the methods for the events that it is interested in. The proxy
495 checks each receiver to see if it has implemented the method, and if so, calls the method. All receivers can be
496 forced to implement all methods by setting `errorOnMethodNotFound=True`.
497 
498  from Lang.Events.Proxy import EventProxy, EventReceiver
499  proxy = EventProxy(errorOnMethodNotFound=False)
500 
501  class Foo(EventReceiver):
502  def someEvent(self, parametersOfEvent, moreParams=None):
503  print("I was called")
504 
505  foo = Foo()
506  proxy.addReceiver(foo)
507  proxy.someEvent("abc", moreParams=123)
508 
509 #### Handling uncaught exceptions
510 
511 When an uncaught exception happens, a special `notifyException` method will be called on each EventReceiver
512 automatically, if the method is implemented, with the exception instance and a traceback instance as parameters.
513 What you do with these parameters is up to you, but here is an example:
514 
515  class Foo(EventReceiver):
516  def notifyException(self, exceptionInstance, tracebackInstance):
517  import traceback
518  tracebackStr = "".join(traceback.format_tb(tracebackInstance))
519  exceptionStr = str(exceptionInstance)
520  print(tracebackStr)
521  print()
522  print(exceptionStr)
523 
524 This idea can be combined with event logging, shown below.
525 Also note that `errorOnMethodNotFound` has no effect on `notifyException`, as it is a special case.
526 
527 ### Event logging
528 
529 Python has decent built in logging, but it doesn't follow standard object-oriented concepts where methods represent
530 actions, so the API is not ideal for recording different events in a heavily event based system, as there would need
531 to be a special, separate call to the logging API for every event. The logging API in `Lang.Events.Logging` fixes this.
532 It uses the event handling API shown above, where the `Logging` class is an `EventProxy`, and the loggers are
533 `EventReceiver`s.
534 
535 #### Example using StdoutLogger
536 
537 This is a very simplistic example of logging to stdout:
538 
539  from Lang.Events.Logging import Logging, StdoutLogger
540  log = Logging(StdoutLogger)
541  log.notifyMyEvent("details", "in", "arguments", "here")
542 
543 #### Example using a custom logger and/or multiple loggers
544 
545 When using multiple loggers, the function you call on the `Logging` instance will be called on every logger.
546 
547  from Lang.Events.Logging import Logging, LoggerAbstract, StdoutLogger
548  class MyFileLog(LoggerAbstract):
549  def __init__(self, filePath):
550  super(MyLogger, self).__init__()
551  self._filePath = filePath
552  def notifyFolderCheck(self, folder):
553  with open(self._filePath, "a") as file_:
554  file_.write("Checking folder: " + folder)
555 
556  log = Logging((StdoutLogger, MyFileLog))
557  log.notifyFolderCheck("folder/path/here")

604 ### An improved ArgParser
605 
606 In addition to the plethora of features in [python's built-in argparse module](https://docs.python.org/2.7/library/argparse.html), a few more are added in here:
607 
608 * Improved help formatting, similar to `man`
609 * Addition of 3 way booleans (`True`, `False`, `None`) and automatic handling of any other iterable type
610 * Required named parameters - the built in ArgParser only supports required positional arguments and optional named parameters
611 
612 The new `ArgParser` uses the same interface as the old one, so see the built in `ArgParser` documentation.
613 
614 Example:
615 
616  from Lang.ArgParser import ArgParser
617  parser = ArgParser(argument_default=None, add_help=True, description="Adds a user to a linux machine")
618  parser.add_argument("username")
619  parser.add_argument("-p", "--password", required=False, help="Prompt for password if this is not given")
620  parser.add_argument("-H", "--create-home", type=Bool3Way, required=True,
621  help="Controls home directory creation for user. None uses the default behavior which varies between machines.")
622  args = parser.parse_args()
